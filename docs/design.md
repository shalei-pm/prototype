## 页面会话拆分

建议按以下会话拆分：

1. 起始默认签到页
   - 未签到扫码态
   - 已签到返回态

2. 首次到院建档页
   - 基础必填信息
   - 更多信息领券
   - 信息跳过 / 完整填写发券状态

3. 项目菜单
   - 新购项目
   - 我的资产
   - 支付后返回资产或继续购买

4. 治疗项目确认页
   - 有本次治疗项目
   - 无本次治疗项目
   - 安排面诊（系统分配）/ 无需面诊直接治疗 / 选择服务项目

5. 面诊人员确认页（历史保留）
   - `08` 原型已从当前业务流程中删除，暂不考虑
   - 评审画布继续保留页面，仅作历史方案参考，不进入当前主流程

6. 流程确认页
   - 皮肤检测
   - 术前术后对比照
   - 饮品零食选择

7. 治疗签署文书流程
   - 会话式确认
   - 文档阅读
   - 用户签署
   - 终止签署

8. 自助签到完成
   - 面诊结束进入
   - 治疗签署完成进入
   - 突出完成状态与下一步房间指引
   - 倒计时退出登录并返回起始默认签到页

## 画布与设备

- 统一以小米 Pad 竖屏为原型画布。
- 小米 Pad 常见真实物理分辨率：
  - 小米 Pad 6：横屏 `2880 x 1800`，竖屏 `1800 x 2880`。
  - 小米 Pad 7：横屏 `3200 x 2136`，竖屏 `2136 x 3200`。
- 后续页面原型以“小米 Pad 6 竖屏”为基准逻辑画布：`900 x 1440`。这是按常见 `2x` 设备像素比从 `1800 x 2880` 折算出的设计坐标。
- `520 x 832` 只能作为浏览器里的缩略展示尺寸，不能作为真实设计坐标。若需要在浏览器里并列展示，应将 `900 x 1440` 的 Pad 画布整体缩放到约 `0.58`，而不是直接在 `520 x 832` 上排版。
- 所有页面保持统一画布、统一页面边距、统一按钮尺度、统一字体层级。
- 页面设计必须考虑前台自助场景：客户站在门店前台快速操作，文字要清晰，按钮要大，路径要短。

### 原型尺寸统一红线

- 所有 Pad 高保真页面必须共享同一套设备壳和缩略展示标准：源画布 `900 x 1440`，浏览器缩略展示约 `scale(0.58)`。
- 当前统一预览外框尺寸为 `559 x 872`，对应 CSS 变量建议为 `--pad-preview-w: 559px`、`--pad-preview-h: 872px`、`--pad-frame: 32px`；不要再使用单页自定义的 `522 x 835` 等预览尺寸。
- 不允许某个页面因为内容特殊而单独改变 Pad 壳尺寸、缩放比例、预览高度、设备圆角、屏幕裁切方式。
- 页面差异只能发生在真实 Pad 屏幕内部，不能改变外层原型单元的宽高体感。
- 新页面必须优先复用当前原型文件里的共享结构：`.pad-shell`、`.pad-screen`、`.task-page`、`.topbar`、`.task-content`、`.bottom-bar`。
- 除非团队明确决定升级全局规范，否则不要为单个页面新建一套独立头部、独立底部栏、独立页面骨架，例如单独写一套 `xxx-header / xxx-footer / xxx-picker` 去绕开统一结构。
- 如果确实需要新增页面专用样式，只能约束在该页面的内容区模块内；头部高度、底部操作区高度、页面边距、字体层级仍需跟全局一致。
- 整合评审画布里的每张图必须保持同一尺寸体系；单个页面不能看起来比其他页面更高、更宽、更像另一个设备。

## Pad 应用设计规范补充

本节来自对 Android Large Screens / Material、Microsoft Windows touch target、WCAG 2.2 触控目标等公开规范的转译，用于约束后续 Pad 页面原型。

### 核心原则

- Pad 内页不能简单放大手机 UI。首页可以有海报和品牌尺度，内页必须回到任务型界面，优先阅读效率、扫读效率、操作效率。
- 竖屏 Pad 仍然有足够横向空间，表单、列表、人员选择、签署流程应优先考虑双栏、列表详情、主内容 + 辅助面板，而不是所有内容单列堆叠。
- 页面布局依据当前画布可用空间设计，不按“某个具体硬件名”硬判断；即便当前以小米 Pad 为基准，也要保持可迁移的比例和组件规则。
- 视觉层级主要靠留白、分隔线、浅灰区块、固定底部操作区建立，不靠把内页标题做得很大。
- 页面模块要让用户一眼知道：当前在做什么、必须完成什么、下一步在哪里、能否退出。

### 尺寸与触控

- 所有可点击控件的触控热区不小于 `44 x 44px`；主操作、底部按钮、人员卡片、项目卡片建议不小于 `48px` 高。
- 图标按钮可视觉上是 `20-24px` 图标，但外层点击热区仍需达到 `44-48px`。
- 相邻可点元素之间至少留 `8px` 间距，避免站立操作时误触。
- 表单输入框、列表项、筛选 Tab、签署步骤项建议使用 `48-56px` 的稳定高度，避免 hover / active / 错误提示导致布局跳动。
- 底部主操作区固定、清晰、可触达；危险动作如“终止签署”“登出”要与主流程按钮保持足够距离和视觉区分。

### 间距与栅格

- 使用 4px / 8px 基础栅格，常用间距为 `8 / 12 / 16 / 20 / 24 / 32`。
- Pad 内页外边距建议 `24px` 左右；复杂内容区可用 `16-20px` 内边距。
- 卡片和浅灰模块之间保持一致间距，避免页面出现“每块都像独立海报”的割裂感。
- 业务页面少用大面积装饰图、渐变、巨型 slogan；图片只服务于识别、项目、医生、签署文件等真实内容。

### 字号层级

- 以下字号以 `900 x 1440` 逻辑画布为准，不是 `520 x 832` 缩略画布。
- 内页主标题建议 `24-28px`，只用于页面标题或当前任务；非常复杂的流程页可降到 `22px`。
- 模块标题建议 `17-20px`。
- 列表标题、项目名、人员名、步骤名建议 `15-17px`。
- 正文、说明、标签、表单辅助信息建议 `13-15px`。
- 按钮文字建议 `15-17px`，用字重和颜色区分优先级，不靠超大字号。
- 中文字重必须克制：正文 `400-500`，模块标题 `600-700`，页面主标题最高 `700`。除品牌海报和极少数数字强调外，禁止使用 `800 / 900 / 950` 这类过粗字重。
- 在 `520 x 832` 缩略展示里，上述字号会随画布整体缩放，不应直接用未缩放的 `24px` 标题放在 520 宽画布上。
- 首页海报和欢迎语可以更大，但内页标题不能超过首页主视觉标题的体感；内页禁止出现“傻大、傻粗”的标题和按钮。

### 适合本项目的 Pad 布局模式

- 首次到院建档：左侧基础必填，右侧更多信息领券；底部固定“完成建档 / 跳过更多信息”。
- 项目菜单：顶部轻 Tab，左侧项目分类、适应症或筛选，右侧项目列表 / 我的资产；已选项目使用固定底部摘要和购买按钮。
- 项目购买流程归属于 `04` 的状态变体。`04-A` 是项目选择/确认页，把 `04` 已选的治疗方案拆为具体可购买品相；确认后直接进入 `04-B` 扫码支付。倒计时归零后先查询最终订单状态，确认未支付再关闭订单并进入 `04-D 支付未完成`；支付成功进入 `04-C`，5 秒后自动前往 `05-B 我的资产`。原支付确认半弹窗以 `历史-02` 保留，不再进入主流程。
- 支付成功进入 `05 我的资产` 后，新购项目置顶、标记 `刚刚购买`，并默认选入今日治疗；用户仍可在资产列表中取消选择或继续加入其他资产项目。支付二维码只能使用明确标注为演示、不可支付的本地资源，不能在评审原型中放置真实收款码。
- 治疗项目确认：页面标题使用 `治疗项目确认`；有预约项目时摘要标题使用 `已预约X个项目`。项目卡片沿用项目菜单 / 我的资产中的已选项目卡片样式，并使用整页通栏布局，确认页内不再展示“可直接加入”等资产选择动作。`AI羊点评` 位于项目列表末尾、固定按钮上方，由 AI 根据本次项目动态归纳，内容长度允许变化；输出至少覆盖预计时间计算、改善点汇总和术前注意三类信息。点评采用连续自然段与行内小标题，不使用等宽列、固定单元格或表格形态；服务意图通过底部主按钮推进。选择 `安排面诊（系统分配）` 只表达需要面诊，不进入人员列表；点击后立即后台发起分配与排队，用户同时继续流程确认。系统按项目适配、门店排班和实时排队自动分配。
- 治疗项目确认页左侧项目列表下方可展示 `建议本次一同治疗` 区域。推荐项卡片样式必须与上方已预约项目卡片保持一致，并明确说明 `一同治疗理由`；如果项目已在用户资产中，右侧使用轻量绿色加号按钮；如果项目未在资产中，右侧使用购买按钮。该区块是辅助推荐，不应抢占已预约项目卡片和底部主流程按钮的视觉优先级。
- 治疗项目确认页的项目卡片不展示 `来源` 字段，避免用户把确认页理解成资产 / 新购来源追踪页；卡片信息优先展示治疗时长、剩余次数或规格、流程要求、治疗重点、一同治疗理由等决策信息。
- 治疗项目确认页的无项目状态也应参考有项目状态的顶部与底部骨架，但不生成项目推荐、可加入项目或 AI 羊点评，因为系统没有本次项目作为判断依据。中间区域只表达当前无治疗项目，以及进入项目菜单或安排面诊的下一步入口。
- 流程确认页：使用门店人员逐项询问的问答式表达，不采用冰冷的配置清单，也不设置独立流程摘要侧栏。每项由自然语言问题、用途解释、轻量直角选项和预计额外耗时组成；皮肤检测、术前术后对比照使用 `需要 / 暂不需要`，饮品小食直接平铺可选内容并标明 `不增加时间`。页面保持连续排版，不做卡片套卡片。顶部时间汇总不使用圆角胶囊或营销标签，采用 `基础治疗时长 + 附加服务时长 = 本次预计总时长` 的信息结构，最终总时长为视觉重点，并随选项实时更新。
- 历史原型 `历史-04 面诊人员确认页` 已从当前业务流程中删除，暂不考虑。现有医生单列卡片、执业信息、排队人数、预计等待时间和系统自动安排等设计仅保留作历史评审参考，不再作为当前流程的设计约束或待完善项。
- 治疗签署文书流程：阅读页左侧只用于区分本次待签署的不同文书，使用约 `150px` 的窄栏导航，不展示“会话确认 / 文件阅读 / 用户签署”等流程步骤；右侧完整预览当前文书并占据主要宽度。文书的 `已阅读 / 待读` 由系统根据阅读进度生成，使用状态圆点或对勾表达，不能使用看起来可手动勾选的复选框；当前阅读文件另用绿色左线或背景表示。底部固定 `合并签署` 与 `逐份签署`。如果文件包中存在需要抄写指定文案等特殊签署要求，`合并签署` 置灰，并明确提示因当地卫健委要求无法合并签署，请逐份签署。
- 签名页按第三方签署能力的最小形态设计，但必须显示当前文书名称和 `第X份 / 共Y份`。空白签名状态禁用清除和提交；检测到签名轨迹后启用提交。点击终止签署时在当前页面弹出二次确认，不增加新的主流程节点；评审画布使用 `10-A / 10-B / 10-C` 展示空白、已签名、终止确认三个状态变体。
- 结束提示页：顶部沿用统一通栏，但不设置页面标题。主体以大尺寸完成图形和 `已完成自助签到` 为第一视觉，明确说明服务要求已收到。完成文案下方用小票式列表复述用户已确认的就诊项目、系统自动分配的面诊医生和服务流程，不使用表格或独立卡片；随后突出下一步房间指引，并提示打开新氧青春App查看排队、面诊与房间变化等服务进度。若后台分配尚未完成，页面显示安排中并允许结束签到，后续通过 App 更新，不应阻塞用户。底部左侧为 `再逛逛`，右侧为 `退出登录（10）秒后自动登出`，不再展示复杂步骤列表和独立倒计时模块。

### 调研来源

- Android Developers：大屏与自适应布局建议使用可用窗口空间、窗口尺寸分级，而不是硬按设备判断。参考：`https://developer.android.com/develop/ui/compose/layouts/adaptive/support-different-display-sizes`
- Android Developers / Material：Canonical layouts 包括 list-detail、feed、supporting pane，适合 Pad 上的列表、项目、人员、文件场景。参考：`https://developer.android.com/design/ui/mobile/guides/layout-and-content/canonical-layouts`
- Android Accessibility：触控目标建议至少 `48 x 48dp`，相邻目标建议 `8dp` 以上间距。参考：`https://support.google.com/accessibility/android/answer/7101858`
- Microsoft Learn：触控目标需要兼顾触摸输入的不精确性，一般建议不低于约 `7.5mm` 的可点击区域。参考：`https://learn.microsoft.com/en-us/windows/apps/develop/input/guidelines-for-targeting`
- WCAG 2.2：指针输入目标至少要有基本尺寸或足够间距；重要控件应采用更严格的目标尺寸。参考：`https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html`
- 微信支付：支付结束时间不能早于下单后 1 分钟；超过支付结束时间后用户无法继续支付，商户应在确认订单状态后调用关单接口。参考：`https://pay.weixin.qq.com/doc/v3/merchant/4012791897`

## 视觉风格

- 页面真实 UI 以白色、浅灰为大面积背景。
- 荧光绿和黑色只作为品牌识别、按钮、状态、重点信息点缀。
- 不使用大面积黑底做业务页面背景；黑底主要用于流程图、说明图或强品牌视觉片段。
- 风格关键词：干净、时尚、轻盈、杂志感、到店服务感。
- 色彩不要变成单一绿色页面，要保留白、灰、黑的层次。

推荐基础颜色：

- 荧光绿：`#9cff00`
- 微信小程序绿：`#07c160`
- 黑：`#111111`
- 主文字：`#171717`
- 次级文字：`#69707c`
- 分隔线：`#e5e7eb`
- 页面底：`#ffffff` / `#f7f7f8`

## 品牌与文案

- 海报英文品牌标签统一使用：`Soyoung clinic`
- 中文品牌主标题可使用：`新氧青春`
- 门店示例：`新氧青春北京保利店`
- 姓名示例：`沙磊`
- 称呼示例：`沙大爷`
- 页面文案优先直接、短句、少解释。
- 不在真实业务页面里写“这是功能说明”“这里可以如何使用”这类说明性文案，除非它是用户必须看到的操作提示。

## 页面结构规则

- 左侧导航用于团队查看页面清单，不属于 Pad 真实页面。
- 右侧 Pad 画布才是真实客户界面。
- 一个页面如果存在关键状态差异，应在原型容器里并列展示多个状态，例如：
  - 状态 A：未签到扫码态
  - 状态 B：已签到返回态
- 系统判断不画成真实页面。
- 用户真实看到或操作的界面才画成高保真页面。
- 流程、判断、识别结果用箭头、连接线、线框按钮表达；页面和页面组用模块表达。

### 线框与高保真关系

- 用户提供的手绘图、线框图只代表信息结构、业务逻辑和模块优先级，不代表最终视觉语言。
- 不要照搬线框里的粗黑边框、蓝色选中态、表格线、随手画的比例、临时按钮样式。
- 从线框进入高保真时，必须重新套用本文的设计标准：白灰大底、黑绿点缀、浅灰模块、克制字号、统一圆角、统一间距、统一底部操作区。
- 线框中的“顶部 / 左侧 / 内容 / 底部”关系可以保留，但视觉上应转译成真实 Pad 应用界面，而不是线框的美化版。

## 组件规则

### 顶部与海报

- 起始默认签到页上半屏是一整张海报图，不拆成卡片。
- Pad 页面里的海报本身不做圆角，只由设备屏幕圆角裁切。
- 起始页海报左上角放菜单 icon，可直接进入品相选择菜单。
- 海报物料后续可替换为真实品牌形象、门店视觉、热推品相或活动海报。

### 按钮

- 主操作按钮可以使用黑底绿字或绿底黑字，但要克制。
- 次级按钮使用白底、浅色描边、绿色或黑色文字。
- 页面级大按钮统一使用直角，包括固定底部操作区的主按钮、次按钮，以及首页已签到态的主要页面入口。卡片内部轻操作、状态标签、筛选项和图标按钮可按组件语义保留小圆角或圆形。
- 已登录返回状态不设置“查看已约项目”等重复入口；页面中部只展示今日项目与流程定制两项确认状态，底部使用统一直角按钮恢复签到流程。
- 圆角按钮用于清晰命令；不要滥用大胶囊让页面显得过重。

### 业务页统一骨架

- 业务流程页默认使用统一骨架：
  - 顶部栏：承载返回、页面标题/门店上下文、当前用户标识和登出。
  - 内容区：承载当前页面的主要任务模块。
  - 底部固定操作区：承载主操作、次操作、已选摘要或流程推进。
- 顶部栏和底部栏的高度、边距、字体层级要与同文件其他业务页保持一致。
- 底部操作区必须固定吸底，不跟随内容滚动；内容多时滚动发生在内容区内部。
- 主按钮、次按钮、摘要条使用统一组件尺度；不要为单个页面重新创造一套按钮高度、栏高或阴影体系。
- 页面标题不应超过内页标准标题体感；不能因为内容复杂而把标题、Tab、按钮放大到破坏整体标准。

### 首次到院建档专项标准

- “更多信息领券”属于首次到院建档页内的补充信息区，不是独立页面。
- 建档采用渐进展开：首屏优先完成并确认核心身份信息，确认成功后才自动展开下方选填信息；如果核心信息已由系统预填，也必须由用户主动确认后再展开，不能在页面载入时直接把所有内容同时展示。
- 选填信息展开后，底部操作切换为 `暂不填写 / 完成填写并领取代金券`，代金券只在选填区域内作为激励，不与核心建档任务争夺视觉优先级。
- 当展示未完成填写状态时，更多信息项应表达“可下拉选择”，行尾使用 `请选择 ▾`。
- 如果需要表达用户正在选择某一项，应画成下拉已展开状态：激活行 + 下方选项面板，而不是把所有字段画成已填写结果摘要。
- 只有“更多信息完整填写 / 已发券”状态才把每项展示为已选结果。

### 项目菜单专项标准

- 项目菜单不是普通商品货架，也不是线框表格；它是“按赛道、适应症和具体问题类型组织的解决方案选择页”。
- 项目菜单的页面任务标题放在统一顶栏，例如 `选择治疗项目`；内容区不要再单独放一行大标题占空间。
- 顶部赛道 Tab 示例：`护肤 / 抗衰 / 问题肌 / 美学`。Tab 采用轻量分段控件，不使用粗重表格线。
- 顶部赛道 Tab 是“按钮形 Tab”，不是圆角按钮；项目菜单里的赛道 Tab、左侧适应症 Tab、购买按钮、项目方案卡片原则上不做圆角，保持更大气的杂志版式感。
- 项目菜单使用三级导航：第一层是赛道，例如 `护肤 / 抗衰 / 问题肌 / 美学`；第二层是当前赛道下的适应症，例如 `油 / 干 / 黄 / 黑`；第三层是适应症下的具体问题类型。以“油”为例，第三层包括 `单纯性出油 / 脂质氧化 / 脂溢性皮炎 / 毛囊角化异常`。轻度、中度、重度不属于第四级导航，而是当前问题类型下的方案内容分组，用正文小标题、适用状态说明和分隔留白表达，不能做成可点击 Tab、筛选按钮或选中态。
- 左侧为当前赛道下的适应症快速切换，例如 `油 / 干 / 黄 / 黑`。左侧切换应使用统一触控尺寸和浅灰背景，选中态用黑绿点缀，不做圆角胶囊。
- 第三层问题类型放在右侧内容顶部，使用紧凑的标签式 Tab，不做成与顶部赛道同宽的目录导航，也不与左侧适应症混排。选择问题类型后，下面按 `问题标题 → 特征关键词 → 常见成因` 连续排版，帮助用户快速识别“这是我的问题”；说明区不必放在背景卡片里，可搭配真实项目图、皮肤状态图或案例效果图。
- `单纯性出油` 当前治疗方案口径：轻度使用 `水杨酸治疗 / 光子嫩肤`；重度以 `肉毒微滴注射 / 光子嫩肤` 为主要方案，以 `水杨酸治疗 / 药物治疗` 为辅助方案。涉及药物的内容必须提示需由医生评估，不能让用户直接自行购买或选择具体药物。
- 严重程度标题、适应症描述和其下项目组合必须形成一个紧密视觉组。程度标题只使用纯文字行，不增加灰色背景、边框、侧线或独立标题条；标题与第一张项目卡只保留极小间距，同组后续项目卡同样紧密排布，不同严重程度组之间使用明显更大的段落间距。
- 项目菜单中的水杨酸治疗、光子嫩肤、肉毒微滴注射等属于治疗类别，实际价格取决于后续选择的具体品相和规格，因此方案卡只展示 `XXX–XXX元/次`、`XXX–XXX元/疗程` 等价格区间。点击 `选择具体品相` 后进入 `04-A 品相选择确认`，再进入支付确认。
- 品相选择确认页按已选治疗方案分组，例如水杨酸治疗下展示博乐达水杨酸治疗、院线复配水杨酸治疗，光子嫩肤下展示 CC光治疗、BBL Hero治疗。治疗组使用纵向通栏列表，不使用左右双列卡片；组标题前使用方形对钩表示该治疗组是否纳入，组内各具体品相名称前使用圆形单选控件。每个治疗组内只能选择一个具体品相，也允许取消组标题前的对钩以整组暂不选择；全页至少选择一个具体品相，否则 `进入支付确认` 按钮置灰且不能继续。品相行的右侧只重点展示券后实付价、优惠前划线价和系统自动匹配的当前最优优惠券；预计治疗时长放在左侧说明区弱化展示，并增加近 30 天同组选购比例作为决策参考。底部汇总和后续支付订单使用相同的优惠前金额、优惠合计与实付金额口径。
- 项目区标题不要叫“推荐项目”，避免让顾客感到被营销；使用 `可选方案` 这类中性表达。
- 项目卡片是页面主角，必须突出用户决策信息：
  - 项目名称
  - 价格 / 规格
  - 套餐或次数
  - 治疗时长
  - 疼痛度
  - 是否需面诊 / 签署 / 恢复期等关键标签
  - 案例效果图或前后对比图
  - 选择 / 已选状态
- 项目卡片要使用真实业务卡片语言：白底、浅灰信息块、轻分隔、绿黑状态点缀；不照搬线框表格行。
- 项目卡片优先使用横向方案行，而不是 2x2 宫格卡片；每行横向展示项目说明、价格/规格/时长/疼痛度、案例图和选择状态，方便扫读比较。
- 方案行需要保留适应症严重程度分组，例如 `轻度`、`重度`，说明不同程度建议选择什么项目。
- 项目方案行不要使用黑色重边框或圆角卡片；选中态可用绿色侧边线、轻描边或状态标签表达。
- 方案行的已选状态不要做成大面积按钮；使用轻量对钩 icon 或状态点即可。未选中状态不需要展示“选择”两个字，也不要显示黑色背景按钮。
- 项目菜单底部固定吸底展示已选摘要和购买按钮。新购项目底栏左侧展示 `已选 X 个治疗项目`，下方灰字展示 `预计耗时 XX 分钟 · 实际治疗时间以院内实际情况为准`；总价放在摘要区最右侧，价格字号明显大于其他信息；主按钮放最右侧，例如 `去购买 / 去支付`。
- 底部购买栏必须真正吸附在 Pad 屏幕底部，不应因为内容不足而漂在中部。
- 未选择项目时，购买按钮应置灰或弱化；已选择项目后使用主按钮样式。
- “我的资产”与“新购项目”共享同一 Pad 骨架和横向项目卡片语言，但内容重点不同：新购项目突出购买决策，资产页突出可用资产、剩余次数、订单 ID、是否可直接加入本次治疗。
- “我的资产”页不额外增加左侧菜单筛选，项目卡片直接罗列即可；除非后续资产数量大到影响检索效率，否则不要把资产页做成分类货架。
- 资产项目卡片必须包含剩余次数和订单 ID；可补充适用门店、治疗时长、是否需面诊 / 签署等信息。已选态沿用轻量对钩 icon，普通未选中状态不显示大按钮。
- 当选择的资产项目与今日已约项目存在不适合同日治疗、需间隔、需咨询确认等关系时，在底部操作区上方展示提醒警告。提醒应具体说明冲突关系，但视觉不要喧宾夺主。
- 资产页底部主按钮文案使用 `添加至今日治疗`；不要出现 `继续购买` 作为资产页主流程按钮。

### 二维码与小程序码

- 小程序码示意应接近微信小程序码：中心标识、放射点阵、短线、圆形定位点。
- App 二维码示意应接近标准二维码：三定位角、方块矩阵。
- 码外面不加细线边框。
- 二维码区域不单独放“小程序码 / App 二维码”大标题，依靠码样式和下方说明文字识别。

### 分隔与卡片

- 起始页扫码区保持平铺背景，不使用圆角卡片。
- 两个二维码入口之间用中间细线区隔。
- 页面模块可以用分隔线、浅灰背景、留白建立层级。
- 避免卡片套卡片。

### 登出

- 默认首页不是客户会话页，不放登出按钮。
- 从扫码成功进入业务流程后，每个业务页面都必须有“登出”。
- 登出在业务页右上角统一使用图标按钮样式，参考建档页右上角；不要在顶栏写成 `退出` 文本链接。
- 登出图标左侧固定展示不可点击的当前用户标识，采用 `称谓（手机号末四位）` 格式，例如 `沙先生（7321）`，便于院内人员直接通过 Pad 识别当前用户。用户姓名尚未获取但手机号已知时显示 `用户（7321）`；手机号未获取时只显示已知称谓并提示手机号待补。
- 用户标识保持单行且不做胶囊标签，统一使用 `18px / 700` 强化院内识别；页头空间不足时优先压缩或截断副标题，不能压缩用户标识和登出热区。
- 项目菜单的 `项目选择 / 我的资产` 两个 Tab 同时承担页面标题，不再额外显示“项目菜单”标题。Tab字号与其他业务页标题统一为 `26px`，仅通过字重、颜色和绿色下划线区分当前页；该页头顺序为 `返回｜项目选择/我的资产｜用户标识｜登出`。
- `辅助-01` 已登录返回状态在海报右上角显示纯图标登出按钮，不增加文字；主体欢迎语使用用户称谓，底部保留退出登录按钮。
- 点击“登出”统一结束客户会话，并返回起始默认签到页。

## 首页/扫码签到页当前共识

首页/扫码签到页包含一个主状态和一个辅助状态：

- `01` 首页/扫码签到页：
  - 上半屏整张海报。
  - 下半屏欢迎来到门店。
  - 左侧为小程序码，右侧为 App 二维码；二维码上方分别显示 `小程序签到` 与 `新氧青春App签到`，中间分隔线显示 `or`。
  - 二维码下方分别显示 `使用微信扫码` 与 `使用新氧青春App扫码`。
  - 文案：`请您扫码签到`。

- `辅助-01` 主页/已登录返回状态：
  - 上半屏仍为整张海报。
  - 该状态属于辅助状态，不进入主流程图；仅在已登录客户连续返回主页时出现。
  - 海报右上角显示纯图标登出按钮，与左上角菜单按钮形成对称位置；页面显著位置已经展示用户称谓，因此不重复添加登出文字或用户标识。
  - 下半屏显示 `欢迎 xxx` 和弱提示 `已签到`，欢迎语下方只纵向展示 `今日项目 / 流程定制` 两项确认信息。
  - 两项信息只用于快速查看进度，不分别做成大入口；主按钮自动进入第一个未完成项。
  - 已确认项直接列出本次确认结果，例如今日项目名称；待确认项说明尚缺少的确认内容。两项使用连续纵向排版和分隔线，不使用卡片或按钮式造型。
  - 去掉 `查看已约项目`，避免与今日项目确认产生重复路径。
  - 已签到返回态不再展示品相菜单、医生团队等浏览入口，确保页面只服务于恢复本次签到确认。
  - 底部固定双按钮：左侧 `退出登录`，右侧 `继续确认服务信息`。项目与流程确认均完成但仍有文书等后续任务时，主按钮动态变为 `继续完成签到`；全部完成后直接进入完成页。

## 新会话工作方式

每个页面会话按以下顺序工作：

1. 读取 `docs/design.md`。
2. 读取 `docs/codex-learning-state.md` 中与当前页面相关的章节。
3. 读取当前原型文件中对应页面的 HTML/CSS。
4. 用户讲页面功能和大框架。
5. agent 先复述页面目标、模块、状态、交互。
6. 生成或修改对应页面原型。
7. 用户在浏览器里批注。
8. agent 按批注细调。
9. 页面共识写回 `docs/codex-learning-state.md`，必要时更新本文。

## 对子 agent 的要求

- 只能改自己负责页面的区域。
- 改共享样式前要确认会不会影响其他页面。
- 开始任何页面原型前必须先读取本文，并复用本文定义的统一画布、页面骨架、字号、间距和组件标准。
- 不能因为拿到线框或手绘图就照搬线框视觉；必须把线框转译为高保真设计。
- 不能为单个页面自由发明一套尺寸体系、头部体系、底部栏体系或缩放体系。
- 页面专用样式必须尽量局限在内容区；如果新增全局类名，要说明它是否影响其他页面。
- 不要改变全局视觉方向：白灰为底，绿黑为点缀。
- 不要引入新的大面积颜色主题。
- 不要把系统判断画成页面。
- 不要把默认首页加登出。
- 产出前检查 HTML 能正常解析。
- 产出后说明改了哪些页面、哪些状态、哪些文件。
