# 自助签到台流程沟通状态

更新时间：2026-07-09

内部对齐代码版本：`checkin-flow-consensus-current`

## 当前目标

为连锁门店前台实验性部署一套自助签到台流程。载体为竖屏固定 Pad，客户自助操作，前台仅在需要时辅助讲解，不代操作。最终需要生成一份团队可看的高保真原型流程图，呈现用户自助签到、决定项目、流程安排、服务人员确认、治疗签署文书及结束闭环。

## 视觉与载体

- 载体：Pad，竖屏固定放置。
- 使用者：客户。
- 前台角色：只辅助讲解，不代操作。
- 视觉方向：沿用新氧青春当前海报样式与风格，大气、时尚杂志感。
- 主视觉色：荧光绿偏绿、灰色、黑色。

## 当前整体流程

1. 默认签到页
   - 页面存在两个状态：未签到扫码态、已签到返回态。
   - 上半部分是一整张海报图，展示品牌形象、门店视觉、热推品相或门店活动物料。
   - 海报本身不做圆角，只由 Pad 屏幕圆角裁切。
   - 海报左上角有菜单 icon，点击可直接进入品相选择菜单。
   - 未签到扫码态下，下半部分为扫码签到。
   - 扫码区保持整体背景平铺，不使用圆角边框卡片；左右两个码之间用细线区隔。
   - 二维码区域不单独放“小程序码 / App 二维码”标题，依靠二维码本体和下方文字注解识别。
   - 小程序码示意需要接近微信小程序码的放射点阵和中心标识；App 二维码示意需要接近标准二维码的三定位角与方块矩阵。
   - 左侧小程序，右侧 App。
   - 已签到成功后如果中途回到首页，页面作为签到恢复页：显示“欢迎 xxx”和弱提示“已签到”，并纵向展示今日项目、服务人员、流程定制三项核心信息的完成状态。
   - 已确认项直接列出确认结果，待确认项说明尚缺少的内容；页面不再展示“查看已约项目”、品相菜单或医生团队等旁路入口。底部固定“退出登录 / 继续确认服务信息”，主按钮进入第一个未完成的确认项。

2. 客户识别
   - 扫码后判断新客或老客。
   - 客户识别是系统判断，不是页面节点；流程图中不画独立页面卡片，也不单独出现登出。
   - 新客进入首次到院建档。
   - 如果建档过程中发现其实是老客，切回“老客直达项目确认”的流转。
   - 老客进入“本次治疗项目确认页”。

3. 新客建档流程
   - 首次到院建档。
   - “首次到院建档”是一个页面节点。
   - 基础必填信息：姓名、身份证号、希望如何称呼、手机号。
   - 身份证需要校验。
   - 手机号优先自动获取 UID 信息；UID 未获取到时在此补填。
   - 更多信息可填写以获取 50 元新客券，包括：
     - 年龄范围选择
     - 医美经验选择
     - 做过与接受医美项目
     - 获知渠道选择
     - 医美诉求选择
     - 执业类型选择
     - 居住城市选择
   - 更多信息允许跳过，跳过后仍可完成建档并进入项目选择，但不发 50 元新客券。
   - 如果更多信息全部填写完成，优惠券区域显示对钩水印，并发放 50 元新客券。
   - “更多信息领券”和“建档完成”只是首次到院建档页面内的补充说明，不是独立页面，不单独出现登出。
   - 新客建档完成后直接进入“项目菜单”。
   - 新客完成项目选择或购买后，进入“本次治疗项目确认页”。

4. 老客直达流转
   - 老客默认进入“本次治疗项目确认页”。
   - 预约只是治疗项目来源之一，不再把这个页面命名为“预约确认页”。
   - 流程图表达上，识别为老客后只保留流转箭头，直接落到“本次治疗项目确认页”，不再画老客说明框。

5. 本次治疗项目确认页
   - 这是新客和老客共同流入的页面。
   - 新客从“项目菜单”完成选择或购买后进入。
   - 老客扫码识别后默认进入。
   - 流程图表达上，识别为新客的箭头落到“项目菜单”；识别为老客的箭头直接落到“本次治疗项目确认页”；“项目菜单”再通过向右箭头指向“本次治疗项目确认页”。
   - 页面按本次是否已有治疗项目展示不同内容和按钮。
   - 有本次治疗项目：展示本次治疗项目列表，并展示预计治疗时长。
   - 有项目时页面底部有两个按钮：
     - 左侧：安排面诊
     - 右侧：无需面诊，直接治疗
   - 无本次治疗项目：提示尚未预约或尚未选择服务项目。
   - 无项目时页面底部有两个按钮：
     - 左侧：安排面诊
     - 右侧：选择服务项目
   - 点击“选择服务项目”后进入“项目菜单”。
   - 如果已有项目但用户需要增加服务项目，也进入“项目菜单”。
   - 完成确认后进入流程确认页，并带入用户选择的服务意图。

6. 项目菜单
   - 这是可复用模块。
   - 新客建档完成后必进。
   - 老客在已有项目主动增加服务项目，或无项目点击“选择服务项目”时进入。
   - 模块包含两个 Tab：
     - 新购项目：代表用户要新购项目。
     - 我的资产：展示用户之前已购但未到店消费过的项目。
   - 从新购项目选择项目后走下单支付。
   - 点击“去购买”后进入项目菜单内的支付子流程：
     - `04-A` 支付确认：覆盖在项目菜单上的半弹窗，列出购买项目、价格、优惠与应付金额，底部为 `取消 / 支付`。
     - `04-B` 扫码支付：完整任务页，展示应付金额、演示二维码、微信支付 / 支付宝和 60 秒有效期；超时后待支付订单取消。
     - `04-C` 支付成功：5 秒后自动进入 `05 我的资产`，也可点击 `查看我的资产` 立即进入。
     - `04-D` 支付未完成：兼容超时、取消与失败，3 秒后回到 `04 项目菜单-新购项目`。
   - 支付成功进入我的资产后，新购项目置顶并默认选入今日治疗，用户仍可取消选择或继续添加其他资产。
   - 选好项目后回到“本次治疗项目确认页”，继续推进到流程确认。

7. 流程确认页
   - 确认是否皮肤检测。
   - 确认是否拍摄术前术后对比照。
   - 饮品零食选择，依据门店配置展示。
   - 确认后进入服务人员确认页。

8. 面诊人员确认页
   - 面诊人员确认页在流程确认页之后。
   - 系统根据前序业务流程决定本页确认的是面诊医生还是面诊咨询师，并动态替换页面标题和人员列表。
   - 当前评审原型只展示 `面诊医生确认`，不混入治疗师或纯生美判断。
   - 医生使用紧凑的单列横向卡片，一屏应完整展示 5-6 位医生；原型照片只需统一示意，不使用高仿真人像。
   - 卡片展示执业标签、从业年限、短简介和擅长方向，避免因卡片过高降低比较效率。
   - 前方排队人数和预计等待时间是主要决策信息，需要突出展示。
   - 支持系统分配。
   - 也支持选择具体人员。
   - 底部左按钮为 `按我的选择安排`，右按钮为 `系统自动安排（最快）`。
   - 面诊人员确认完成后，直接进入统一结束提示页。

9. 治疗签署文书流程
   - 这是复杂子流程，不是单一签署按钮。
   - 包含：
     - 会话式提问确认
     - 文档阅读
     - 用户签署
   - 用户一次可能需要签署多份文书；阅读页左侧 Tab 只区分不同文件，右侧完整预览当前文件，不再把“会话确认 / 文件阅读 / 用户签署”作为左侧步骤。
   - 当前原型示例文件为 `就诊告知书` 与 `光电治疗知情同意书`，不展示项目风险确认书和术后护理须知。
   - 底部提供 `合并签署` 与 `逐份签署` 两种方式。
   - 如果文件包中存在因当地卫健委要求必须由用户抄写指定文案的特殊文书，`合并签署` 置灰，并提示用户改用逐份签署。
   - 实际签名可能调用第三方能力，当前原型表达标准顶栏、当前文书名称、份数进度、签字面板和 `终止签署返回 / 提交签署` 两个按钮；空白签名时提交置灰，签名轨迹存在后启用提交。
   - 11号签署页用 `11A / 11B / 11C` 展示空白签名、已签名和终止确认三个状态变体，仍属于同一业务页面；终止操作使用页内二次确认层，不增加新的流程节点。
   - 签署流程可以终止。
   - 点击“终止签署”后的回退点待确认；此前共识是返回“安排面诊 / 无需面诊直接治疗”的选择状态。
   - 点击“登出”则直接返回首页。
   - 必签项完成后进入统一结束提示页。

10. 统一结束提示页
    - 需要面诊的：服务人员确认完成后直接进入统一结束提示页。
    - 不需要面诊直接治疗的：完成治疗签署文书后进入统一结束提示页。
    - 页面重点表达 `已完成自助签到，您的服务要求我们已经收到。`
    - 完成文案下方以小票式列表复述本次就诊项目、面诊医生和服务流程，不使用表格。
    - 当前示例指引为 `请您到2号面诊室进行面诊。`
    - 房间指引下方提示用户打开新氧青春App查看服务进度。
    - 顶部沿用统一业务通栏，但不显示页面标题。
    - 底部左按钮为 `再逛逛`，右按钮为 `退出登录（10）秒后自动登出`。
    - 倒计时结束后自动登出并返回默认签到页。
    - 当前建议倒计时为 10 秒，尚未最终确认。

## 全局规则

- 从扫码成功进入业务流程后，每个业务页面都必须有“登出”按钮。
- 点击“登出”后，统一结束当前客户会话，并返回起始默认签到页。
- 默认首页本身不属于客户会话页，不需要同样的登出逻辑。

## 下一阶段高保真原型组织方式

- 后续需要把各页面对应的高保真原型逐步细化出来。
- 原型承载结构建议采用左右布局：
  - 左侧为页面导航，放置各个页面名称或页面组名称。
  - 右侧为细化后的高保真页面原型图。
  - 每个流程模块中贴入对应页面或页面组的高保真原型，方便团队同时看流程位置和页面细节。
- 后续梳理规则：
  - 先出整体流程。
  - 页面和页面组用模块表示。
  - 流程、判断、识别结果用箭头、连接线和线框按钮表示。
  - 系统判断不画成页面节点；用户真实看到或操作的界面才画成页面原型。

## 当前整合原型索引

以下编号以 `prototype/integrated-review-current.html` 为准，后续评审可直接按编号引用：

1. `01` 签到起始页-未扫码
2. `02` 签到起始页-已扫码
3. `03` 建档页
4. `04` 项目菜单-新购项目
   - `04-A` 项目购买-支付确认
   - `04-B` 项目购买-扫码支付
   - `04-C` 项目购买-支付成功
   - `04-D` 项目购买-支付未完成
5. `05` 项目菜单-我的资产（支付成功进入时，新购项目置顶并默认选入今日治疗）
6. `06` 治疗项目确认页
7. `07` 治疗项目确认 · 无项目
8. `08` 服务人员-面诊医生确认页（当前原型仅展示医生；咨询师场景由系统动态替换标题与人员列表）
9. `09` 治疗流程确认页（逐项问答式确认附加服务；顶部按基础时长、附加时长和预计总时长进行汇总）
10. `10` 治疗文书签署-文书列表
11. `11A / 11B / 11C` 治疗文书-用户签署（空白签名、已签名、终止确认三个状态变体，仍属于同一业务页面）
12. `12` 签到分诊完成页

“建档 · 更多信息完整领券”不再作为独立评审原型；领券仍属于首次到院建档页内部规则。
建档页采用渐进展开：用户先确认核心身份信息，确认完成后自动展开选填信息和领券内容；选填阶段允许暂不填写。
治疗项目确认页的 `AI羊点评` 为动态生成的归纳正文，不按固定指标表格排版；输出格式至少包含预计时间计算、改善点汇总和术前注意，具体段落长度随本次项目信息变化。

## 后续待继续补充

- 项目选择菜单的详细交互和状态。
- 本次治疗项目确认页的有项目 / 无项目两种状态布局。
- 流程确认页中皮肤检测、拍照、饮品零食的门店配置规则。
- 服务人员确认页中面诊医生、咨询师、治疗医生的排序、等待时间、系统分配逻辑。
- 治疗签署文书流程中的第三方能力边界、抄写文案校验、提交失败与重试状态。
- 治疗签署终止后的回退点。
- 门店内部履约动线：前台、咨询师、医生、护士 / 治疗师分别何时收到提醒，看到什么任务。
- 异常分流：身份校验失败、支付失败、项目不可用、资产不可用、医生不可约、签署失败、用户中途登出等。
- 最终高保真原型流程图的呈现形式。
