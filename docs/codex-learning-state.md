# 自助签到台流程沟通状态

更新时间：2026-07-09

内部对齐代码版本：`checkin-flow-consensus-current`

## 当前目标

为连锁门店前台实验性部署一套自助签到台流程。载体为竖屏固定 Pad，客户自助操作，前台仅在需要时辅助讲解，不代操作。最终需要生成一份团队可看的高保真原型流程图，呈现用户自助签到、决定项目、流程安排、服务人员确认、治疗签署文书及结束闭环。

## 视觉与载体

- 载体：Pad，竖屏固定放置。
- 使用者：客户。
- 前台角色：只辅助讲解，不代操作。
- 视觉方向：沿用新氧青春当前海报样式与风格，大气、时尚杂志感。
- 主视觉色：荧光绿偏绿、灰色、黑色。

## 当前整体流程

1. 默认签到页
   - 页面存在两个状态：未签到扫码态、已签到返回态。
   - 上半部分是一整张海报图，展示品牌形象、门店视觉、热推品相或门店活动物料。
   - 海报本身不做圆角，只由 Pad 屏幕圆角裁切。
   - 海报左上角有菜单 icon，点击可直接进入品相选择菜单。
   - 未签到扫码态下，下半部分为扫码签到。
   - 扫码区保持整体背景平铺，不使用圆角边框卡片；左右两个码之间用细线区隔。
   - 二维码区域不单独放“小程序码 / App 二维码”标题，依靠二维码本体和下方文字注解识别。
   - 小程序码示意需要接近微信小程序码的放射点阵和中心标识；App 二维码示意需要接近标准二维码的三定位角与方块矩阵。
   - 左侧小程序，右侧 App。
   - 已签到成功后如果回到首页，页面下半部分变为“欢迎 xxx女士”，右上角弱提示“已签到”，并在右侧提供轻量“查看已约项目”按钮；左侧入口为品相菜单，右侧入口为医生团队。

2. 客户识别
   - 扫码后判断新客或老客。
   - 客户识别是系统判断，不是页面节点；流程图中不画独立页面卡片，也不单独出现登出。
   - 新客进入首次到院建档。
   - 如果建档过程中发现其实是老客，切回“老客直达项目确认”的流转。
   - 老客进入“本次治疗项目确认页”。

3. 新客建档流程
   - 首次到院建档。
   - “首次到院建档”是一个页面节点。
   - 基础必填信息：姓名、身份证号、希望如何称呼、手机号。
   - 身份证需要校验。
   - 手机号优先自动获取 UID 信息；UID 未获取到时在此补填。
   - 更多信息可填写以获取 50 元新客券，包括：
     - 年龄范围选择
     - 医美经验选择
     - 做过与接受医美项目
     - 获知渠道选择
     - 医美诉求选择
     - 执业类型选择
     - 居住城市选择
   - 更多信息允许跳过，跳过后仍可完成建档并进入项目选择，但不发 50 元新客券。
   - 如果更多信息全部填写完成，优惠券区域显示对钩水印，并发放 50 元新客券。
   - “更多信息领券”和“建档完成”只是首次到院建档页面内的补充说明，不是独立页面，不单独出现登出。
   - 新客建档完成后直接进入“项目菜单”。
   - 新客完成项目选择或购买后，进入“本次治疗项目确认页”。

4. 老客直达流转
   - 老客默认进入“本次治疗项目确认页”。
   - 预约只是治疗项目来源之一，不再把这个页面命名为“预约确认页”。
   - 流程图表达上，识别为老客后只保留流转箭头，直接落到“本次治疗项目确认页”，不再画老客说明框。

5. 本次治疗项目确认页
   - 这是新客和老客共同流入的页面。
   - 新客从“项目菜单”完成选择或购买后进入。
   - 老客扫码识别后默认进入。
   - 流程图表达上，识别为新客的箭头落到“项目菜单”；识别为老客的箭头直接落到“本次治疗项目确认页”；“项目菜单”再通过向右箭头指向“本次治疗项目确认页”。
   - 页面按本次是否已有治疗项目展示不同内容和按钮。
   - 有本次治疗项目：展示本次治疗项目列表，并展示预计治疗时长。
   - 有项目时页面底部有两个按钮：
     - 左侧：安排面诊
     - 右侧：无需面诊，直接治疗
   - 无本次治疗项目：提示尚未预约或尚未选择服务项目。
   - 无项目时页面底部有两个按钮：
     - 左侧：安排面诊
     - 右侧：选择服务项目
   - 点击“选择服务项目”后进入“项目菜单”。
   - 如果已有项目但用户需要增加服务项目，也进入“项目菜单”。
   - 完成确认后进入流程确认页，并带入用户选择的服务意图。

6. 项目菜单
   - 这是可复用模块。
   - 新客建档完成后必进。
   - 老客在已有项目主动增加服务项目，或无项目点击“选择服务项目”时进入。
   - 模块包含两个 Tab：
     - 新购项目：代表用户要新购项目。
     - 我的资产：展示用户之前已购但未到店消费过的项目。
   - 从新购项目选择项目后走下单支付。
   - 支付后可返回资产列表，也可继续在项目列表购买。
   - 选好项目后回到“本次治疗项目确认页”，继续推进到流程确认。

7. 流程确认页
   - 确认是否皮肤检测。
   - 确认是否拍摄术前术后对比照。
   - 饮品零食选择，依据门店配置展示。
   - 确认后进入服务人员确认页。

8. 服务人员确认页
   - 服务人员确认页在流程确认页之后。
   - 根据前序业务流程与用户选择，分两栏展示需要确认的服务人员类型。
   - 左栏：面诊咨询师 / 医生。
   - 右栏：治疗师。
   - 如果是纯生美场景，跳过治疗师选择。
   - 页面需要展示当前在店人员、简介、等待状态、预计等待时间。
   - 支持系统分配。
   - 也支持选择具体人员。
   - 选择完成后按服务类型分流：
     - 需要面诊的：服务人员确认完成后，直接进入统一结束提示页。
     - 不需要面诊直接治疗的：进入治疗签署文书流程。

9. 治疗签署文书流程
   - 这是复杂子流程，不是单一签署按钮。
   - 包含：
     - 会话式提问确认
     - 文档阅读
     - 用户逐项签署
     - 完成校验
   - 签署流程可以终止。
   - 点击“终止签署”后的回退点待确认；此前共识是返回“安排面诊 / 无需面诊直接治疗”的选择状态。
   - 点击“登出”则直接返回首页。
   - 必签项完成后进入统一结束提示页。

10. 统一结束提示页
    - 需要面诊的：服务人员确认完成后直接进入统一结束提示页。
    - 不需要面诊直接治疗的：完成治疗签署文书后进入统一结束提示页。
    - 提示用户按照 App 或小程序中的服务流程指引继续。
    - 页面进入倒计时登出。
    - 倒计时结束后自动登出并返回默认签到页。
    - 当前建议倒计时为 10 秒，尚未最终确认。

## 全局规则

- 从扫码成功进入业务流程后，每个业务页面都必须有“登出”按钮。
- 点击“登出”后，统一结束当前客户会话，并返回起始默认签到页。
- 默认首页本身不属于客户会话页，不需要同样的登出逻辑。

## 下一阶段高保真原型组织方式

- 后续需要把各页面对应的高保真原型逐步细化出来。
- 原型承载结构建议采用左右布局：
  - 左侧为页面导航，放置各个页面名称或页面组名称。
  - 右侧为细化后的高保真页面原型图。
  - 每个流程模块中贴入对应页面或页面组的高保真原型，方便团队同时看流程位置和页面细节。
- 后续梳理规则：
  - 先出整体流程。
  - 页面和页面组用模块表示。
  - 流程、判断、识别结果用箭头、连接线和线框按钮表示。
  - 系统判断不画成页面节点；用户真实看到或操作的界面才画成页面原型。
