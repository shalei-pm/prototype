# 自助签到台流程沟通状态

更新时间：2026-07-09

内部对齐代码版本：`checkin-flow-consensus-current`

## 当前目标

为连锁门店前台实验性部署一套自助签到台流程。载体为竖屏固定 Pad，客户自助操作，前台仅在需要时辅助讲解，不代操作。最终需要生成一份团队可看的高保真原型流程图，呈现用户自助签到、本次项目确认、流程安排、治疗签署文书及结束闭环。服务人员确认暂不纳入当前流程。

## 视觉与载体

- 载体：Pad，竖屏固定放置。
- 使用者：客户。
- 前台角色：只辅助讲解，不代操作。
- 视觉方向：沿用新氧青春当前海报样式与风格，大气、时尚杂志感。
- 主视觉色：荧光绿偏绿、灰色、黑色。

## 当前整体流程

1. 首页/扫码签到页
   - `01` 是主流程中的扫码入口；`辅助-01` 是已登录用户连续返回主页时出现的辅助状态，不进入主流程图。
   - 上半部分是一整张海报图，展示品牌形象、门店视觉、热推品相或门店活动物料。
   - 海报本身不做圆角，只由 Pad 屏幕圆角裁切。
   - 海报左上角有菜单 icon，点击可直接进入品相选择菜单。
   - 未签到扫码态下，下半部分为扫码签到。
   - 扫码区保持整体背景平铺，不使用圆角边框卡片；左右两个码之间用细线区隔。
   - 二维码上方分别显示 `小程序签到` 与 `新氧青春App签到`，中间分隔线显示 `or`；二维码下方分别显示 `使用微信扫码` 与 `使用新氧青春App扫码`。
   - 小程序码示意需要接近微信小程序码的放射点阵和中心标识；App 二维码示意需要接近标准二维码的三定位角与方块矩阵。
   - 左侧小程序，右侧 App。
   - 已签到成功后如果中途回到首页，页面作为签到恢复页：显示“欢迎 xxx”和弱提示“已签到”，并纵向展示今日项目、流程定制两项确认信息；服务人员不再出现在该页面。
   - 已确认项直接列出确认结果，待确认项说明尚缺少的内容；页面不再展示“查看已约项目”、品相菜单或医生团队等旁路入口。底部固定“退出登录 / 继续确认服务信息”，主按钮进入第一个未完成的确认项。

2. 客户识别
   - 扫码后判断新客或老客。
   - 客户识别是系统判断，不是页面节点；流程图中不画独立页面卡片，也不单独出现登出。
   - 新客进入首次到院建档。
   - 如果建档过程中发现其实是老客，切回“老客直达项目确认”的流转。
   - 老客进入“本次治疗项目确认页”。

3. 新客建档流程
   - 首次到院建档。
   - “首次到院建档”是一个页面节点。
   - 基础必填信息：姓名、身份证号、希望如何称呼、手机号。
   - 身份证需要校验。
   - 手机号优先自动获取 UID 信息；UID 未获取到时在此补填。
   - 更多信息可填写以获取 50 元新客券，包括：
     - 年龄范围选择
     - 医美经验选择
     - 做过与接受医美项目
     - 获知渠道选择
     - 医美诉求选择
     - 执业类型选择
     - 居住城市选择
   - 更多信息允许跳过，跳过后仍可完成建档并进入项目选择，但不发 50 元新客券。
   - 如果更多信息全部填写完成，优惠券区域显示对钩水印，并发放 50 元新客券。
   - “更多信息领券”和“建档完成”只是首次到院建档页面内的补充说明，不是独立页面，不单独出现登出。
   - 新客建档完成后直接进入“项目菜单”。
   - 新客完成项目选择或购买后，进入“本次治疗项目确认页”。

4. 老客直达流转
   - 老客默认进入“本次治疗项目确认页”。
   - 预约只是治疗项目来源之一，不再把这个页面命名为“预约确认页”。
   - 流程图表达上，识别为老客后只保留流转箭头，直接落到“本次治疗项目确认页”，不再画老客说明框。

5. 本次治疗项目确认页
   - 这是新客和老客共同流入的页面。
   - 新客从“项目菜单”完成选择或购买后进入。
   - 老客扫码识别后默认进入。
   - 流程图表达上，识别为新客的箭头落到“项目菜单”；识别为老客的箭头直接落到“本次治疗项目确认页”；“项目菜单”再通过向右箭头指向“本次治疗项目确认页”。
   - 页面按本次是否已有治疗项目展示不同内容和按钮。
   - 有本次治疗项目：展示本次治疗项目列表，并展示预计治疗时长。
   - 有项目时页面底部有两个按钮：
     - 左侧：安排面诊（系统分配）
     - 右侧：无需面诊，直接治疗
   - 无本次治疗项目：提示尚未预约或尚未选择服务项目。
   - 无项目时页面底部有两个按钮：
     - 左侧：安排面诊（系统分配）
     - 右侧：选择服务项目
   - 点击“选择服务项目”后进入“项目菜单”。
   - 如果已有项目但用户需要增加服务项目，也进入“项目菜单”。
   - 完成确认后直接进入流程确认页；服务人员确认页已从当前流程删除。
   - 选择安排面诊只表达服务意图；点击后立即在后台发起人员分配与排队，用户同时继续流程确认。系统根据项目适配、门店排班和实时排队自动分配，不向用户提供指定医生或咨询师入口。

6. 项目菜单
   - 这是可复用模块。
   - 新客建档完成后必进。
   - 老客在已有项目主动增加服务项目，或无项目点击“选择服务项目”时进入。
   - 模块包含两个 Tab：
     - 新购项目：代表用户要新购项目。
     - 我的资产：展示用户之前已购但未到店消费过的项目。
   - 新购项目采用三级内容导航：赛道 → 适应症 → 具体问题类型。以 `护肤 → 油` 为例，问题类型包括 `单纯性出油 / 脂质氧化 / 脂溢性皮炎 / 毛囊角化异常`。
   - 每个问题类型使用标签式 Tab；选中后先展示问题标题和高度概括的特征关键词，再展示常见成因，随后按轻度、重度组织可选治疗方案。严重程度是内容分组，不是第四级导航。
   - `单纯性出油` 示例中，轻度方案为水杨酸治疗、光子嫩肤；重度主要方案为肉毒微滴注射、光子嫩肤，水杨酸治疗与药物治疗作为辅助方案，药物部分需医生评估。
   - 严重程度标题使用纯文字行，不增加背景或边框；标题与同组项目卡紧密排布，不同程度组之间增加段落间距。项目菜单中的治疗类别价格均展示区间，选定具体品相后再展示确定价格。
   - 从新购项目选择治疗方案后，先进入品相选择确认，再进入下单支付。
   - `04-A` 品相选择确认：按已选治疗方案分组展示具体品相，使用纵向通栏列表；治疗组标题前用方形对钩表示整组是否纳入，组内具体品相名称前用圆形单选，每组最多选一项且允许整组取消；全页至少选择一项，否则不能进入支付确认。品相行右侧聚焦券后实付、优惠前划线价和自动匹配的当前最优券，左侧弱化展示预计时长并增加近 30 天同组选购比例；底部与支付页保持同一金额口径。
   - 品相确认后进入项目菜单内的支付子流程：
     - `04-B` 扫码支付：完整任务页，展示应付金额、演示二维码、微信支付 / 支付宝和 60 秒有效期；超时后待支付订单取消。
     - `04-C` 支付成功：5 秒后自动进入 `05-B 我的资产`，也可点击 `查看我的资产` 立即进入。
     - `04-D` 支付未完成：兼容超时、取消与失败，3 秒后回到 `05-A 项目菜单`。
     - `历史-02` 保留原支付确认半弹窗，不再进入当前主流程。
   - 支付成功进入我的资产后，新购项目置顶并默认选入今日治疗，用户仍可取消选择或继续添加其他资产。
   - 选好项目后回到“本次治疗项目确认页”，继续推进到流程确认。

7. 面诊人员确认页（历史原型）
   - `历史-04 服务人员-面诊医生确认页` 已从当前业务流程中删除，暂不考虑。
   - 评审画布继续保留该原型并标注历史状态，仅供回溯此前方案。

8. 流程确认页
   - 流程确认页在本次治疗项目确认后直接进入，用于补充本次到店服务的细节安排。
   - 确认是否皮肤检测。
   - 确认是否拍摄术前术后对比照。
   - 饮品零食选择，依据门店配置展示。
   - 完成后根据此前选择的服务意图分流：需要面诊时读取已并行发起的系统分配结果并进入结束闭环；直接治疗时进入治疗签署文书流程。

9. 治疗签署文书流程
   - 这是复杂子流程，不是单一签署按钮。
   - 包含：
     - 会话式提问确认
     - 文档阅读
     - 用户签署
   - 用户一次可能需要签署多份文书；阅读页左侧 Tab 只区分不同文件，右侧完整预览当前文件，不再把“会话确认 / 文件阅读 / 用户签署”作为左侧步骤。
   - 当前原型示例文件为 `就诊告知书` 与 `光电治疗知情同意书`，不展示项目风险确认书和术后护理须知。
   - 底部提供 `合并签署` 与 `逐份签署` 两种方式。
   - 如果文件包中存在因当地卫健委要求必须由用户抄写指定文案的特殊文书，`合并签署` 置灰，并提示用户改用逐份签署。
   - 实际签名可能调用第三方能力，当前原型表达标准顶栏、当前文书名称、份数进度、签字面板和 `终止签署返回 / 提交签署` 两个按钮；空白签名时提交置灰，签名轨迹存在后启用提交。
   - 签署页用 `10-A / 10-B / 10-C` 展示空白签名、已签名和终止确认三个状态变体，仍属于同一业务页面；终止操作使用页内二次确认层，不增加新的流程节点。
   - 签署流程可以终止。
   - 点击“终止签署”后的回退点待确认；此前共识是返回“安排面诊 / 无需面诊直接治疗”的选择状态。
   - 点击“登出”则直接返回首页。
   - 必签项完成后进入统一结束提示页。

10. 统一结束提示页
    - 需要面诊的：项目确认时已在后台发起人员分配，流程细节确认完成后进入统一结束提示页。
    - 不需要面诊直接治疗的：完成治疗签署文书后进入统一结束提示页。
    - 页面重点表达 `已完成自助签到，您的服务要求我们已经收到。`
    - 完成文案下方以小票式列表复述本次就诊项目、系统分配的面诊医生和服务流程，不使用表格。
    - 当前示例指引为 `请您到2号面诊室进行面诊。`；若此时分配尚未完成，展示 `正在为您安排面诊，请留意新氧青春App通知`，不能阻塞完成签到。
    - 房间指引下方提示用户打开新氧青春App查看服务进度。
    - 顶部沿用统一业务通栏，但不显示页面标题。
    - 底部左按钮为 `再逛逛`，右按钮为 `退出登录（10）秒后自动登出`。
    - 倒计时结束后自动登出并返回默认签到页。
    - 当前建议倒计时为 10 秒，尚未最终确认。

## 全局规则

- 从扫码成功进入业务流程后，每个业务页面都必须有“登出”按钮。
- 右上角登出图标左侧统一展示当前用户标识，格式为 `称谓（手机号末四位）`，当前示例为 `沙先生（7321）`；该标识仅用于院内人员识别，不可点击。
- 04项目选择与05我的资产使用Tab直接承担页面标题，页头结构为 `返回｜项目选择/我的资产｜用户标识｜登出`，不再额外显示“项目菜单”标题。
- `辅助-01` 已登录返回状态在海报右上角增加纯图标登出按钮，不增加文字；主体显示 `欢迎，沙先生`，底部保留退出登录。
- 点击“登出”后，统一结束当前客户会话，并返回起始默认签到页。
- 默认首页本身不属于客户会话页，不需要同样的登出逻辑。

## 下一阶段高保真原型组织方式

- 后续需要把各页面对应的高保真原型逐步细化出来。
- 原型承载结构建议采用左右布局：
  - 左侧为页面导航，放置各个页面名称或页面组名称。
  - 右侧为细化后的高保真页面原型图。
  - 每个流程模块中贴入对应页面或页面组的高保真原型，方便团队同时看流程位置和页面细节。
- 后续梳理规则：
  - 先出整体流程。
  - 页面和页面组用模块表示。
  - 流程、判断、识别结果用箭头、连接线和线框按钮表示。
  - 系统判断不画成页面节点；用户真实看到或操作的界面才画成页面原型。

## 当前整合原型索引

以下编号以 `prototype/integrated-review-current.html` 为准，后续评审可直接按编号引用：

1. `01` 首页/扫码签到页
2. `01-A / 01-B` 新客建档（身份信息核验、完善个人信息）
3. `02-A / 02-B` 签到成功-确认咨询意向（有预约、无预约）
4. `03-A / 03-B` 咨询分诊结果（已分配咨询室、排队中）
5. `04` 方案设计
   - `04-A` 项目选择/确认
   - `04-B` 项目购买-扫码支付
   - `04-C` 项目购买-支付成功
   - `04-D` 项目购买-支付未完成
6. `05-A / 05-B` 项目菜单-自助选购、我的资产
7. `06` 本次治疗确认页
8. `07` 服务要求确认页
9. `08` 治疗方案待确认页（无需咨询分支的值班医生复核状态）
10. `09` 治疗文书签署-文书列表
11. `10-A / 10-B / 10-C` 治疗文书-用户签署（空白签名、已签名、终止确认）
12. `11-A / 11-B` 治疗分诊结果（已分配治疗室、排队中）

辅助与历史页面不再占用主流程编号，并统一排在整合画布末尾：

- `辅助-01` 主页/已登录返回状态
- `历史-01` 原04 方案选择/确认页
- `历史-02` 原04-B 项目购买-支付确认
- `历史-03` 本次治疗确认页 · 无项目
- `历史-04` 服务人员-面诊医生确认页

“建档 · 更多信息完整领券”不再作为独立评审原型；领券仍属于首次到院建档页内部规则。
建档页采用渐进展开：用户先确认核心身份信息，确认完成后自动展开选填信息和领券内容；选填阶段允许暂不填写。
治疗项目确认页的 `AI羊点评` 为动态生成的归纳正文，不按固定指标表格排版；输出格式至少包含预计时间计算、改善点汇总和术前注意，具体段落长度随本次项目信息变化。
